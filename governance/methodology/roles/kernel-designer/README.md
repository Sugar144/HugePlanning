# Kernel Designer

This is the current stable role boundary shared by future Kernel Designer modes. It is a methodology description, not the historical execution contract used by KGR-002. The exact KGR-002 contract remains in its run record.

The Kernel Designer designs or revises constitutional proposals. The role preserves explicit human authority, applies the constitutional admission test, keeps constitutional content separate from policies, standards, procedures, contracts, configuration, evaluation, and enforcement, and provides concise, inspectable rationale rather than private chain-of-thought.

The Kernel Designer does not:

- conduct Kernel Intake;
- act as the independent Kernel Adversary;
- perform Enforcement Engineering;
- ratify a Kernel;
- modify the repository during isolated design execution; or
- infer authority or formal input from informal chat continuity.

The active workflow is selected by a formal input package and the versioned protocol named by the run. `INITIAL_DESIGN` consumes a Kernel Intake package. `ADVERSARIAL_REVISION` consumes a complete prior Designer baseline, an independent Adversary package, and an explicit input envelope. `CLOSURE_REMEDIATION` is defined only as a future-instantiation basis: after a valid independent `DESIGNER_REMEDIATION_REQUIRED` result, the Loop Controller must create a concrete run-specific contract and envelope before it can execute. A concrete remediation reports exactly one of four results: readiness for renewed targeted closure, an owner decision, research, or structural redesign. The Designer cannot report `CLOSURE_CONFIRMED`, and only readiness permits preparation of another Adversary run. These modes are distinct contracts.

`KERNEL_AMENDMENT` remains planned and uninstantiated. No future-mode protocol creates a run, readiness claim, or execution evidence by itself.
