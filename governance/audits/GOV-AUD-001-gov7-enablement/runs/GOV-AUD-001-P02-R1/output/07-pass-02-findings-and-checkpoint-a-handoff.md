## Strongest proposals

**PROPOSAL:** Use eleven ownership-based bounded contexts: constitutional authority, governance derivation, audit custody, methodology definition, methodology runtime, client/project state, agent/skill execution, validation/evaluation, learning/feedback, release/migration and projection/query.

**PROPOSAL:** Keep canonical authority in source artifacts. Treat the typed relationship model, impact analysis, compatibility matrix and any later graph/query representation as derived, regenerable and non-authoritative.

**PROPOSAL:** Use a generic governance-application contract shape, instantiated only for materially governed stages or transitions. Do not create one bespoke contract per folder or one universal workflow engine.

**PROPOSAL:** Maintain a 26-type relationship vocabulary and sixteen query contracts that answer clause-to-control, stage, executor, validator, evidence, gate, owner, compatibility, migration, impact, historical correction and independence questions.

**PROPOSAL:** Require explicit compatibility declarations across eleven version domains. The comparison covers exact lock, compatible range, capability declaration, migration contract, adapter/projection, revalidation and unsupported combination.

**PROPOSAL:** Route upward findings without automatic source mutation. Every finding has one primary canonical owner, evidence, permitted action, escalation path, closure validator and residual-risk authority.

**PROPOSAL:** Controlled self-hosting requires a human-controlled bootstrap trust root, separate proposal/validation/evaluation/acceptance/implementation/release/operation steps, tested rollback/manual fallback and no self-certification or self-ratification.

**RECOMMENDATION:** Keep `NO_DISTRIBUTION_YET` as an explicit decision option until a named capability satisfies all distribution criteria. This is not selection of a final architecture.
